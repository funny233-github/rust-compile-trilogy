# 代码生成论 — 大纲草案

## 定位

纯理论教程。解释「为什么代码能这样翻译」，
从 C→x86 汇编开始建立直觉，逐步推到 Rust→汇编，
最后用这套理论重新审视 proc-macro-guide 和 rust2mlog。

## 角色

每一层抽象都是一个「翻译器」，从高到低：

C 抽象机 → 栈帧 / 寄存器 / 三地址码 → x86 汇编 → 机器码
Rust 抽象机 → MIR → LLVM IR → 机器码
DSL 抽象机 → AST → 三地址码 IR → MLOG

三者在本质上共享同一条翻译链，只是目标不同。

## 章节大纲

### 序章：代码生成的普遍性
- 编译器是什么：一层层翻译，每层丢掉一些信息，添加一些细节
- 三个翻译链并列展示：C→x86, Rust→LLVM, MLOG DSL→MLOG
- 核心问题：高级概念（变量、循环、函数）如何变成低层指令
- 本书结构

### 第一章：最低抽象 — CPU 看到的世界
- 寄存器：CPU 能直接操作的最快存储
- 内存：大但慢，需要 load/store
- 指令：mov / add / cmp / jmp / call / ret
- 无类型：一切都是一串 bit
- 无作用域：只有地址
- 以 x86-64 为参考，但不深入具体指令编码

### 第二章：C 表达式 → 汇编
- 算术表达式：add/sub/imul/idiv
- 三地址码形式：dest = src1 op src2
- 临时值：寄存器和栈槽
- 类型如何消失：int/short/char → 不同宽度的 mov
- 指针解引用：mov rax, [rbx]
- 数组：基址 + 偏移 × 元素大小
- 一个完整示例：C 算术表达式到 x86 的逐条对照

### 第三章：C 控制流 → 汇编
- if/else → cmp + jmp
- while → 标签 + 条件跳转（和 MLOG 一模一样！）
- for → while 展开
- switch → 跳转表 vs 链式比较
- 短路求值：&& 和 || 的跳转模式
- 控制流图（CFG）的基础概念

### 第四章：C 函数 → 汇编
- 调用约定的意义：谁清理栈？参数怎么传？
- 栈帧：rbp / rsp，local variables，返回地址
- call / ret 指令
- cdecl vs System V AMD64 ABI
- 参数传递：寄存器传参 vs 栈传参
- 返回值：rax
- leaf function 优化
- 递归的栈展开

### 第五章：寄存器分配 — 编译器的心脏
- 无限虚拟寄存器 → 有限物理寄存器
- 活跃范围（live range）
- 图着色分配（graph coloring）的直觉
- 溢出（spilling）：当寄存器不够用时
- MLOG 的变量就是寄存器——无限制，无需分配
- 寄存器分配是最难也是最重要的编译优化

### 第六章：三地址码 — 编译器的通用语言
- 为什么需要中间表示
- SSA 形式：每个变量只赋值一次
- phi 函数：在控制流汇合点选择值
- SSA → 非 SSA 的解构
- MLOG 就是三地址码——这就是为什么 MLOG 是理想的编译目标

### 第七章：从 C 到 Rust — 多了什么
- C 编译器已经解决的问题
- Rust 多了什么：所有权、借用、生命周期、trait、enum
- 但大部分在编译时解决——运行时和 C 一样
- monomorphization：泛型 = 编译时复制粘贴
- 动态分发：vtable 和 fat pointer
- enum → tagged union：tag + union 的内存布局

### 第八章：Rust 特有构造 → 汇编
- `Option<T>` 的 niche 优化：`None` 可以不占额外空间
- `Result<T, E>` 的布局
- pattern match → 条件跳转链（或跳转表）
- 闭包：struct + 函数指针（无捕获）或 struct + vtable（有捕获）
- `Box<T>`：就是指针
- `.await`：状态机展开
- 这些 Rust 构造的编译结果和手工 C 的汇编惊人地相似

### 第九章：优化 — 编译器在帮你做什么
- 常量折叠和传播
- 死代码消除
- 公共子表达式消除（CSE）
- 循环不变量外提
- 内联
- 尾调用优化
- 强度削减
- 优化的整体哲学：不改变语义，只改善性能

### 第十章：从通用理论到 MLOG
- MLOG 的底层模型回顾
- 为什么 MLOG 编译器和 x86 编译器本质相同
- 差异在哪：无寄存器压力、无调用栈、无限变量
- 差异导致的设计简化：不需要寄存器分配、不需要栈帧
- 但核心骨架完全一致：AST → 三地址码 → 线性指令
- 这就是为什么用 1000 行就能写出 MLOG 编译器

### 第十一章：到过程宏 — 为什么选择编译时
- 运行时 vs 编译时代码生成的权衡
- 过程宏提供的是"编译器前端框架"
- 任何「源语言 → 目标语言」的翻译都可以用 proc macro 实现
- shader 编译、SQL 生成、协议解析、配置文件验证
- 理论指导实践：当你理解了翻译链，就知道什么适合放进过程宏，什么不适合

### 终章：翻译链的普遍性
- 所有编译器都是翻译器
- 从 C 到 x86、从 Rust 到 LLVM、从 DSL 到 MLOG——同构
- 理解了这三条链，你就理解了编译器设计的核心
- 推荐阅读 + 继续探索的方向
